Power BI Overview

Core concepts · Interview essentials · BA perspective

What is Power BI?

Power BI is Microsoft's business intelligence tool that lets you connect to data sources, transform and model data, build visualizations, and share reports with stakeholders. As a BA, you'll use it to turn raw data into dashboards that help decision-makers act faster.

The 3 Core Components

ComponentWhat it isWhen you use it
Power BI DesktopFree Windows app where you build reports and data modelsDay-to-day development — connecting data, writing DAX, designing visuals
Power BI ServiceCloud platform (app.powerbi.com) where you publish and share reportsPublishing finished reports, setting up dashboards, managing workspace access
Power BI GatewayA bridge that lets the Service reach on-premises data sourcesWhen your data lives on a company SQL Server or file share, not in the cloud
Interview tip: Interviewers often ask "what is the difference between a report and a dashboard?" — this trips people up. A report is built in Desktop and can have multiple pages with detailed visuals. A dashboard lives only in the Service and is a single canvas of pinned tiles from one or more reports. You cannot create a dashboard in Desktop.

Key Concepts to Know

TermPlain-language explanation
DatasetThe published data model in the Service — the tables, relationships, and measures your report pulls from
DataflowA reusable ETL process defined in the Service using Power Query. Think of it as a shared data prep layer multiple reports can use, instead of each report connecting and transforming data independently
Power QueryThe data transformation layer inside Desktop (also called "M" language). You use it to clean, reshape, and combine data before it loads into your model
WorkspaceA folder in the Service where reports, datasets, and dashboards live. Teams are organized into workspaces with role-based access

Common Interview Questions

Q: What is Power BI and how have you used it?
Sample answer: "Power BI is Microsoft's BI platform. I've used Desktop to build reports on top of SQL Server and Snowflake data, written DAX measures for KPIs, and published reports to the Service for business stakeholders to access through dashboards."

Q: What is the difference between a dataset and a dataflow?
Sample answer: "A dataset is the published data model a report connects to. A dataflow is an upstream data preparation step — it transforms and stores data in the Service so multiple datasets can reuse the same cleaned data without duplicating transformation logic."

Q: What is Power Query used for?
Sample answer: "Power Query is used to clean and shape data before it enters the data model — removing columns, filtering rows, merging tables, changing data types, and handling nulls. It runs before your DAX calculations."

Practical tip: Even if you haven't built Power BI reports from scratch, frame your experience around the outcome — "I validated the data model," "I defined the KPI requirements," "I did UAT on the dashboard" — that is legitimate BA involvement.

DAX Formulas

Measures · Calculated columns · Key functions · Interview Q&A

What is DAX?

DAX (Data Analysis Expressions) is the formula language used in Power BI to create calculations. You write DAX to define measures (like Total Sales) and calculated columns (like a Profit Margin % column). It looks similar to Excel formulas but is designed to work on entire tables and relationships, not individual cells.

Measure vs Calculated Column — A Classic Interview Question

MeasureCalculated Column
What it isA formula that calculates at query time based on filter contextA new column added to a table, calculated row by row when data loads
Stored inNot stored — computed on the flyStored in the model, increases file size
When to useAggregations, KPIs, anything that changes with slicers/filtersRow-level values you want to use in filters, slicers, or relationships
ExampleTotal Revenue = SUM(Sales[Amount])Full Name = Employees[First] & " " & Employees[Last]
Interview tip: The most common wrong answer is "use a calculated column for totals." Always ask yourself: does this value change depending on what filters are applied? If yes → use a measure.

Core DAX Functions

CALCULATE — Evaluates an expression in a modified filter context. The most important DAX function.

DAX 
-- Total sales for only the "North" region, regardless of what region slicer is selected
North Sales = CALCULATE(SUM(Sales[Amount]), Sales[Region] = "North")

SUMX — Iterates row by row over a table, evaluates an expression per row, then sums the results. Use when you need to multiply before summing.

DAX 
-- SUM(Sales[Revenue]) would fail if Revenue isn't a column.
-- SUMX lets you calculate it row by row first.
Total Revenue = SUMX(Sales, Sales[Quantity] * Sales[UnitPrice])

FILTER — Returns a filtered table. Usually used inside CALCULATE or SUMX.

DAX 
-- Sales only where the order amount exceeded $1,000
High Value Sales = CALCULATE(
    SUM(Sales[Amount]),
    FILTER(Sales, Sales[Amount] > 1000)
)

IF — Standard conditional logic.

DAX 
-- Flag orders as large or small
Order Size = IF(Sales[Amount] > 500, "Large", "Small")

RELATED — Pulls a value from a related table (like a VLOOKUP across a relationship). Only works in calculated columns, not measures.

DAX 
-- In the Sales table, bring in the Category name from the Products table
Product Category = RELATED(Products[Category])

Time Intelligence Functions

These only work if your model has a proper Date table with a continuous date range marked as a Date Table.

TOTALYTD — Year-to-date cumulative total.

DAX 
Revenue YTD = TOTALYTD(SUM(Sales[Amount]), Dates[Date])

SAMEPERIODLASTYEAR — Returns the same period from the prior year, used for year-over-year comparisons.

DAX 
Revenue LY = CALCULATE(
    SUM(Sales[Amount]),
    SAMEPERIODLASTYEAR(Dates[Date])
)
Practical tip: You don't need to memorize every DAX function. What interviewers care about is whether you understand the difference between measures and calculated columns, and whether you can explain CALCULATE — because CALCULATE is the engine behind almost every advanced DAX pattern.
Common mistake: Writing a measure that uses RELATED. RELATED only works in calculated columns because it needs a row context. In a measure, use LOOKUPVALUE instead.

Data Modeling

Star schema · Relationships · Fact vs dimension · Cardinality

Star Schema vs Snowflake Schema

A star schema has one central fact table connected directly to multiple dimension tables. It looks like a star. A snowflake schema normalizes those dimension tables further — dimensions connect to sub-dimensions. Star schema is preferred in Power BI because it's simpler for DAX and performs better.

Text 
STAR SCHEMA                        SNOWFLAKE SCHEMA

   DimDate                            DimDate
      |                                  |
DimProduct — FactSales — DimCustomer  DimProduct — FactSales — DimCustomer
                |                         |                        |
           DimRegion              DimCategory              DimCity
                                                              |
                                                          DimCountry

Flat, fast, easy to query          More normalized, more joins needed
Interview tip: When asked "what data model do you prefer in Power BI and why?" — say star schema. Explain that it reduces the complexity of DAX relationships and improves query performance because Power BI's engine (VertiPaq) is optimized for it.

Fact Tables vs Dimension Tables

Fact TableDimension Table
ContainsMeasurable events and transactions — sales, orders, log entriesDescriptive attributes — customer info, product details, dates
RowsMany — one row per transactionFewer — one row per entity (one per customer, one per product)
ColumnsForeign keys + numeric measures (Amount, Quantity)Primary key + descriptive text columns (Name, Category, Region)
ExampleFactSales: OrderID, CustomerID, ProductID, Date, AmountDimCustomer: CustomerID, Name, City, Segment

Relationships and Cardinality

Cardinality describes how many rows in one table match rows in another. Power BI lets you define:

Relationship TypeWhat it meansExample
One-to-Many (1:*)One row in Table A matches many rows in Table B. The most common and preferred type.One Customer → Many Orders
Many-to-One (*:1)Same as one-to-many, just from the other directionMany Orders → One Customer
One-to-One (1:1)Each row in A matches exactly one row in B. Rare — usually means you can merge the tables.EmployeeID → EmployeeDetails
Many-to-Many (*:*)Multiple rows in A match multiple rows in B. Avoid if possible — causes ambiguous filter propagation.Students ↔ Courses (without a bridge table)

Cross-Filter Direction

This controls how filters flow between tables. Single means filters flow from the one-side to the many-side only (recommended). Both means filters flow in both directions — can cause unexpected results in DAX and performance issues. Use "Both" sparingly and only when you understand the impact.

Common mistake — Many-to-Many relationships: If you see a many-to-many in a model, the clean solution is to introduce a bridge table that breaks it into two one-to-many relationships. For example, instead of Students ↔ Courses directly, create an Enrollment table (StudentID, CourseID) and connect each side as one-to-many.
Practical tip: In interviews, if asked to critique a data model, look for: missing date table, many-to-many relationships, calculated columns where measures should be used, and dimensions not connected to the fact table.

Visuals & Reports

Chart selection · Interactivity · RLS · Interview Q&A

Choosing the Right Visual

VisualBest used forAvoid when
Bar / Column chartComparing categories (sales by region, count by department)You have more than ~10 categories — gets crowded
Line chartTrends over time (monthly revenue, daily ticket volume)You're comparing categories without a time axis
Donut / Pie chartPart-to-whole when you have 3–5 segments maxYou have many segments or need exact comparisons — use a bar chart instead
MatrixTabular data with row and column groupings (like a pivot table)Users need to export raw data — a table visual is simpler
CardSingle KPI values (Total Revenue: $4.2M)You need context or trend — pair with a sparkline or use KPI visual
Scatter chartCorrelation between two numeric measuresNon-technical stakeholders — harder to read
SlicerUser-controlled filtering (date range, region dropdown)You have too many values — use a search slicer or dropdown mode
Interview tip: If asked "why would you use a matrix instead of a table?" — a matrix supports row and column groupings with subtotals and conditional formatting by group. A table is flat. Matrices are better for pivot-style reporting.

Key Interactivity Features

Slicers — Panel filters that users interact with directly on the report page. They filter all visuals on the page by default. You can sync slicers across pages for consistency.

Drill-through — A right-click navigation feature that lets users jump from a summary visual to a detail page, carrying filter context with them. Example: right-click on "North Region" in a bar chart → drill through to a detail page showing all North Region orders.

Bookmarks — Saved states of a report page (which visuals are visible, which filters are applied). Used to build toggle buttons, guided story navigation, or "reset filters" buttons without any code.

Drill-down — Moving down a hierarchy within the same visual. Example: Year → Quarter → Month → Day in a line chart. Different from drill-through (which navigates to a new page).

Row-Level Security (RLS)

RLS restricts which rows of data a user can see when they view a report. You define roles in Desktop (e.g., "North Region Manager sees only rows where Region = 'North'"), then assign users to those roles in the Service.

As a BA, you need to understand RLS because:

  • You'll gather requirements for who should see what data
  • You'll define the rules with the data team or validate them during UAT
  • Stakeholders will ask why their report shows different numbers than a colleague's — RLS is often the answer
Common mistake: Confusing drill-through with drill-down. Drill-down stays on the same visual and moves through a hierarchy. Drill-through navigates to a separate report page. In interviews, be precise about which one you mean.

Common Interview Questions

Q: What is the difference between a filter and a slicer?
A filter is applied in the Filters pane and is not visible to the report user by default. A slicer is a visual element on the canvas that the user can interact with directly.

Q: What is RLS and why does it matter?
RLS controls data visibility at the row level based on the logged-in user. It matters for compliance, confidentiality, and ensuring users only see data relevant to their role.

Q: How would you allow a manager to see only their team's data?
Define an RLS role with a DAX filter on the relevant table (e.g., [ManagerEmail] = USERPRINCIPALNAME()), then assign that manager's account to the role in the Service.

SQL Overview

What SQL is · DDL vs DML vs DCL · Query execution order · Key concepts

What is SQL?

SQL (Structured Query Language) is the standard language for working with relational databases. As a BA, you use SQL to query data directly from databases like SQL Server, IBM DB2, or Snowflake — pulling the data you need to analyze, validate, or hand off to stakeholders without needing a developer to run every query for you.

DDL vs DML vs DCL

CategoryStands forWhat it coversExamples
DDLData Definition LanguageStructure of the database — creating and modifying tablesCREATE, ALTER, DROP, TRUNCATE
DMLData Manipulation LanguageThe data inside tables — reading and changing recordsSELECT, INSERT, UPDATE, DELETE
DCLData Control LanguagePermissions and accessGRANT, REVOKE

As a BA, you'll mostly use DML — especially SELECT. You rarely write DDL or DCL unless you're setting up a reporting database.

Query Execution Order — Why It Matters

SQL doesn't execute in the order you write it. The database engine processes clauses in this order:

Text 
1. FROM       — which table(s) to use
2. JOIN       — combine tables
3. WHERE      — filter rows (before grouping)
4. GROUP BY   — group the remaining rows
5. HAVING     — filter groups (after grouping)
6. SELECT     — choose which columns to return
7. ORDER BY   — sort the result
8. LIMIT/TOP  — restrict number of rows returned

This is why you cannot use a column alias in a WHERE clause — the alias is defined in SELECT (step 6), but WHERE runs at step 3. By the time WHERE runs, the alias doesn't exist yet.

Interview tip: Interviewers love asking "why can't you use a column alias in a WHERE clause?" The answer is execution order — WHERE runs before SELECT. You CAN use an alias in ORDER BY because ORDER BY runs after SELECT.

Key Concepts

ConceptPlain explanation
Primary KeyA column (or combination of columns) that uniquely identifies every row in a table. Cannot be NULL, cannot be duplicated. Example: EmployeeID in an Employees table.
Foreign KeyA column in one table that references the primary key of another table. It creates the link between tables. Example: Orders.CustomerID references Customers.CustomerID.
IndexA database structure that speeds up data retrieval — like a book index. Indexing a column you frequently filter or join on makes queries faster. As a BA, you don't create indexes but should know to ask the DBA if a slow query might benefit from one.
NULLThe absence of a value — not zero, not an empty string. NULL means unknown. You must use IS NULL or IS NOT NULL to check for it; using = NULL never returns results.
SchemaA namespace or container that organizes database objects. In SQL Server: dbo.Employees. In Snowflake: database.schema.table.

Common Interview Questions

Q: What is the difference between a primary key and a foreign key?
A primary key uniquely identifies a row within its own table. A foreign key is a reference to the primary key of another table — it's how you create a relationship between two tables.

Q: What is an index and when would you use one?
An index speeds up data retrieval on a column. You'd add one to columns frequently used in WHERE clauses or JOINs. The trade-off is that indexes slow down INSERT/UPDATE operations slightly.

Q: What databases have you worked with?
"I've written queries against SQL Server and IBM DB2 for operational reporting, and Snowflake for cloud data warehouse work. The syntax is largely standard SQL with minor dialect differences."

SELECT & Filtering

SELECT · WHERE · LIKE · IN · BETWEEN · NULL handling · ORDER BY

Basic SELECT

SQL 
-- Get all columns from the employees table
SELECT * FROM employees;

-- Get specific columns only (preferred in production — never use * in reports)
SELECT employee_id, first_name, last_name, department, salary
FROM employees;

WHERE — Filtering Rows

SQL 
-- Single condition
SELECT * FROM employees WHERE department = 'Finance';

-- Multiple conditions with AND
SELECT * FROM employees
WHERE department = 'Finance' AND salary > 70000;

-- OR condition
SELECT * FROM employees
WHERE department = 'Finance' OR department = 'HR';

-- NOT condition
SELECT * FROM employees
WHERE NOT department = 'Finance';

LIKE — Pattern Matching

% matches any sequence of characters. _ matches exactly one character.

SQL 
-- Names starting with 'Ja'
SELECT * FROM employees WHERE first_name LIKE 'Ja%';

-- Names ending with 'son'
SELECT * FROM employees WHERE last_name LIKE '%son';

-- Names with exactly 5 characters
SELECT * FROM employees WHERE last_name LIKE '_____';

-- Email addresses from a specific domain
SELECT * FROM employees WHERE email LIKE '%@company.com';

IN, BETWEEN, DISTINCT

SQL 
-- IN: match any value in a list (cleaner than multiple OR conditions)
SELECT * FROM employees
WHERE department IN ('Finance', 'HR', 'Operations');

-- BETWEEN: inclusive range (includes both endpoints)
SELECT * FROM employees
WHERE salary BETWEEN 50000 AND 80000;

-- DISTINCT: remove duplicate values
SELECT DISTINCT department FROM employees;
-- Returns each department name once, no matter how many employees are in it

NULL Handling

SQL 
-- Find employees with no manager assigned
SELECT * FROM employees WHERE manager_id IS NULL;

-- Find employees who do have a manager
SELECT * FROM employees WHERE manager_id IS NOT NULL;

-- WRONG — this will never return any rows:
SELECT * FROM employees WHERE manager_id = NULL;  -- Don't do this!
Common mistake — = NULL: WHERE column = NULL always returns zero rows because NULL is not equal to anything, not even itself. Always use IS NULL or IS NOT NULL.

ORDER BY and LIMIT / TOP

SQL 
-- Sort by salary descending, then name ascending
SELECT employee_id, first_name, salary
FROM employees
ORDER BY salary DESC, first_name ASC;

-- Top 10 highest paid employees (SQL Server / Snowflake syntax)
SELECT TOP 10 first_name, salary
FROM employees
ORDER BY salary DESC;

-- Same result in standard SQL / Snowflake / PostgreSQL
SELECT first_name, salary
FROM employees
ORDER BY salary DESC
LIMIT 10;
Practical tip: In Snowflake and most modern databases, use LIMIT. In SQL Server, use TOP. In IBM DB2, use FETCH FIRST 10 ROWS ONLY. Knowing these dialect differences will come up if your interviewer asks about specific databases you've worked with.

JOINs

INNER · LEFT · RIGHT · FULL OUTER · SELF JOIN · NULL behavior

What is a JOIN?

A JOIN combines rows from two tables based on a related column. You'll use JOINs constantly as a BA to combine data from multiple tables — for example, pulling employee names from one table and their department names from another.

The examples below use two tables: employees (emp_id, name, dept_id) and departments (dept_id, dept_name).

INNER JOIN — Only matching rows from both tables

SQL 
SELECT e.name, d.dept_name
FROM employees e
INNER JOIN departments d ON e.dept_id = d.dept_id;
-- Returns only employees who have a matching dept_id in departments.
-- Employees with no dept_id (NULL) are excluded.
-- Departments with no employees are excluded.

LEFT JOIN — All rows from the left table, matching rows from the right

SQL 
SELECT e.name, d.dept_name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.dept_id;
-- Returns ALL employees.
-- If an employee has no matching department, dept_name will be NULL.
-- Use LEFT JOIN when you want to keep all records from the left (first) table.

RIGHT JOIN — All rows from the right table, matching rows from the left

SQL 
SELECT e.name, d.dept_name
FROM employees e
RIGHT JOIN departments d ON e.dept_id = d.dept_id;
-- Returns ALL departments, even those with no employees.
-- If a department has no employees, e.name will be NULL.
-- In practice, most analysts just flip the table order and use a LEFT JOIN instead.

FULL OUTER JOIN — All rows from both tables

SQL 
SELECT e.name, d.dept_name
FROM employees e
FULL OUTER JOIN departments d ON e.dept_id = d.dept_id;
-- Returns ALL employees AND ALL departments.
-- Where there's no match, NULLs fill in on the non-matching side.
-- Useful for finding records that exist in one table but not the other.

SELF JOIN — Joining a table to itself

SQL 
-- Find each employee and their manager's name
-- (assuming manager_id references emp_id in the same table)
SELECT e.name AS employee, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.emp_id;
-- You must alias the table twice to treat it as two separate tables.
Visual reference — JOIN types: 
INNER JOIN         LEFT JOIN          RIGHT JOIN         FULL OUTER JOIN
  [A ∩ B]          [A + A∩B]          [B + A∩B]           [A + A∩B + B]
   (A)(B)           (A)(B)             (A)(B)               (A)(B)
    overlap         left+overlap       right+overlap        everything
    only            only               only

Common Interview Questions

Q: What is the difference between INNER JOIN and LEFT JOIN?
INNER JOIN returns only rows with a match in both tables. LEFT JOIN returns all rows from the left table plus matching rows from the right — unmatched right-side columns come back as NULL.

Q: What happens to NULLs in a JOIN?
NULL does not equal NULL, so rows where the join key is NULL will never match. In an INNER JOIN they're dropped. In a LEFT JOIN, the left-side row is kept but the right-side columns will be NULL.

Q: When would you use a FULL OUTER JOIN?
When you need to find discrepancies between two tables — for example, finding orders that exist in a staging table but not in production, or vice versa. Also useful in data reconciliation work.

Aggregates & GROUP BY

COUNT · SUM · AVG · GROUP BY rules · HAVING vs WHERE

Aggregate Functions

FunctionWhat it doesNote
COUNT(*)Counts all rows in the group, including NULLsUse when you want a row count regardless of column values
COUNT(column)Counts non-NULL values in that columnIf 5 out of 10 rows have a NULL phone number, COUNT(phone) = 5
SUM(column)Adds up all numeric values (ignores NULLs)SUM(salary) across a department
AVG(column)Average of non-NULL valuesNote: AVG ignores NULLs, so it may not reflect the true average if NULLs represent zero
MIN / MAXSmallest or largest valueWorks on dates too — MIN(order_date) gives the earliest order

Example 1 — Simple COUNT

SQL 
-- How many employees are in the company?
SELECT COUNT(*) AS total_employees FROM employees;

-- How many employees have a phone number on file?
SELECT COUNT(phone_number) AS employees_with_phone FROM employees;

Example 2 — GROUP BY with SUM

Every column in SELECT that is NOT inside an aggregate function must appear in GROUP BY. This is the most common GROUP BY mistake.

SQL 
-- Total salary cost per department
SELECT department, SUM(salary) AS total_salary_cost
FROM employees
GROUP BY department;
-- 'department' is in SELECT and is not aggregated, so it must be in GROUP BY.

-- Multiple grouping columns
SELECT department, job_title, COUNT(*) AS headcount, AVG(salary) AS avg_salary
FROM employees
GROUP BY department, job_title;

Example 3 — GROUP BY with HAVING

HAVING filters after grouping. WHERE filters before grouping. You cannot use an aggregate function in a WHERE clause — that's what HAVING is for.

SQL 
-- Departments with more than 10 employees
SELECT department, COUNT(*) AS headcount
FROM employees
GROUP BY department
HAVING COUNT(*) > 10;

-- Wrong — this will throw an error:
SELECT department, COUNT(*) AS headcount
FROM employees
WHERE COUNT(*) > 10   -- Can't use aggregate in WHERE
GROUP BY department;

-- Combining WHERE and HAVING
-- Active departments with more than 5 employees
SELECT department, COUNT(*) AS headcount
FROM employees
WHERE status = 'Active'      -- filters rows before grouping
GROUP BY department
HAVING COUNT(*) > 5;         -- filters groups after grouping
Interview tip: The question "what is the difference between WHERE and HAVING?" is almost guaranteed in a SQL interview. WHERE filters individual rows before grouping. HAVING filters groups after GROUP BY has been applied. HAVING can use aggregate functions; WHERE cannot.
Common mistake: Including a column in SELECT that isn't in GROUP BY and isn't aggregated. For example: SELECT department, first_name, COUNT(*) FROM employees GROUP BY department — this fails because first_name is neither aggregated nor in GROUP BY. Most databases will throw an error; some (older MySQL) will return an arbitrary value.

Window Functions

ROW_NUMBER · RANK · DENSE_RANK · LAG · LEAD · Running totals

What is a Window Function?

A window function performs a calculation across a set of rows related to the current row — without collapsing those rows into a single result like GROUP BY does. Each row keeps its own identity; the window function just adds an extra calculated column alongside it.

The syntax always includes an OVER() clause, which defines the "window" of rows to calculate across.

SQL 
function_name() OVER (
    PARTITION BY column   -- optional: restart calculation for each partition
    ORDER BY column       -- required for ranking/running total functions
)

Example 1 — ROW_NUMBER, RANK, DENSE_RANK

All three assign a number to rows based on ORDER BY — but handle ties differently. This is a classic interview question.

SQL 
SELECT name, salary,
    ROW_NUMBER() OVER (ORDER BY salary DESC) AS row_num,
    RANK()       OVER (ORDER BY salary DESC) AS rank_num,
    DENSE_RANK() OVER (ORDER BY salary DESC) AS dense_rank_num
FROM employees;

/* Result:
name       salary   row_num   rank_num   dense_rank_num
Alice      90000       1          1            1
Bob        85000       2          2            2
Carol      85000       3          2            2      <- tie: same rank
Dave       70000       4          4            3      <- RANK skips 3, DENSE_RANK doesn't
*/
Interview tip — RANK vs DENSE_RANK: RANK skips numbers after a tie (1, 2, 2, 4). DENSE_RANK does not skip (1, 2, 2, 3). ROW_NUMBER always assigns a unique number — ties are broken arbitrarily. Use DENSE_RANK when you want a true "position" ranking with no gaps.

Example 2 — PARTITION BY (Rank within groups)

SQL 
-- Rank employees by salary within each department
SELECT name, department, salary,
    RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS dept_rank
FROM employees;

/* Result:
name     department   salary   dept_rank
Alice    Finance      90000       1
Bob      Finance      85000       2
Carol    HR           80000       1      <- rank restarts for each department
Dave     HR           75000       2
*/

Example 3 — LAG and LEAD (Compare to previous/next row)

SQL 
-- Month-over-month revenue comparison
SELECT month, revenue,
    LAG(revenue, 1)  OVER (ORDER BY month) AS prev_month_revenue,
    LEAD(revenue, 1) OVER (ORDER BY month) AS next_month_revenue,
    revenue - LAG(revenue, 1) OVER (ORDER BY month) AS month_over_month_change
FROM monthly_sales;

/* LAG looks back 1 row. LEAD looks forward 1 row.
   The first row has NULL for LAG; the last row has NULL for LEAD. */

Example 4 — SUM OVER (Running Total)

SQL 
-- Cumulative (running) total of revenue by month
SELECT month, revenue,
    SUM(revenue) OVER (ORDER BY month) AS running_total
FROM monthly_sales;

/* Result:
month    revenue   running_total
Jan      10000         10000
Feb      12000         22000
Mar       9000         31000
*/
Practical tip: Window functions are one of the most valuable SQL skills for a BA. They let you do things like "rank stores by revenue within each region," "calculate month-over-month change," or "find each customer's most recent order" — without needing subqueries or GROUP BY that collapses your rows.
Common mistake: Trying to use a window function in a WHERE clause. Window functions are evaluated after WHERE, so you cannot filter on them directly. Wrap your query in a subquery or CTE, then filter on the result. Example: WITH ranked AS (...) SELECT * FROM ranked WHERE dept_rank = 1.

AWS Overview for BAs

What AWS is · Key services · IAM · Cloud vs on-prem · Interview Q&A

What is AWS?

Amazon Web Services (AWS) is a cloud computing platform — a collection of on-demand infrastructure and services (storage, databases, computing, machine learning, etc.) that organizations use instead of running their own physical servers. As a BA, you won't manage AWS infrastructure, but you'll regularly work with data that lives in AWS — in storage buckets, databases, or data warehouses — and you'll collaborate with data engineers who build pipelines on top of it.

Region and Availability Zone

A region is a physical geographic location where AWS has data centers (e.g., us-east-1 for Northern Virginia, eu-west-1 for Ireland). An availability zone (AZ) is an isolated data center within a region — AWS spreads infrastructure across multiple AZs for resilience. As a BA, the main reason this matters is data residency: if your organization must keep data in a specific country for compliance, you'll need to confirm the correct AWS region is being used.

What is IAM?

IAM (Identity and Access Management) is AWS's system for managing permissions. Instead of giving a person direct access to everything, you create roles (a set of permissions) and policies (rules that define what those permissions allow or deny). A user or service assumes a role when they need access.

As a BA, you care about IAM because: when you can't access a dataset or an S3 bucket, the root cause is almost always an IAM policy. Understanding the concept helps you ask the right questions — "Do I have the correct role assigned?" or "Does the Glue job's role have permission to write to this S3 bucket?"

Common Interview Questions

Q: What is the difference between cloud and on-premises?
On-premises means your company owns and maintains its own servers in its own data centers. Cloud means you rent infrastructure from a provider like AWS — you pay for what you use, and AWS handles the hardware maintenance. Cloud is more scalable and flexible; on-prem gives more direct control.

Q: What is S3?
S3 (Simple Storage Service) is AWS's object storage service. You store files in "buckets." It's not a database — you can't run SQL against it directly — but it's the most common place raw data lands before being processed by ETL pipelines.

I queried Redshift using standard SQL for reporting and used Athena for quick ad-hoc exploration of raw S3 data. I've also worked within IAM access policies to understand and troubleshoot data permissions."

Amazon S3

Object storage · Buckets · Storage classes · Data pipeline role · Access control

What is S3?

Amazon S3 (Simple Storage Service) is a cloud object storage service. Think of it as a highly scalable file system in the cloud — you store files (called objects) inside containers (called buckets). S3 is not a database. You cannot run a SQL query directly against S3 in the same way you would a table — the data lives there as files (CSV, JSON, Parquet, etc.) waiting to be processed.

Almost every AWS data pipeline starts and ends with S3. Raw data lands here, ETL processes it, and processed results are often written back to a different S3 location before being loaded to a data warehouse.

Buckets and Objects

S3 has different storage tiers based on how frequently data is accessed. You don't need to know every tier, but two are worth knowing for interviews:

Access Control

Access to S3 is controlled in two ways: Bucket policies (resource-based rules attached to the bucket itself — who can read or write) and IAM policies (user or role-based rules — what that person or service is allowed to do). In practice, both work together. If you can't access a file in S3, either your IAM role doesn't have permission, or the bucket policy is blocking you.

Interview tip — "How did you use S3 in your data work?"
Sample answer: "In our data pipeline, raw data files from source systems were landed in S3 buckets organized by date. AWS Glue jobs would pick up those files, apply transformations, and write the cleaned output to a processed S3 prefix, which was then loaded into Redshift. I worked with S3 paths when writing requirements for Glue jobs and validated the output files during testing."
Practical tip: S3 paths are case-sensitive. "Raw/" and "raw/" are different prefixes. This is a common source of bugs in ETL pipelines — something to flag in your requirements documentation when specifying file paths.

AWS Glue & ETL

What ETL is · Glue crawlers · Data Catalog · BA's role · Interview Q&A

What is ETL?

ETL stands for Extract, Transform, Load — the process of moving data from source systems into a target system (usually a data warehouse) in a usable format.

StepWhat happensExample
ExtractPull data from the source — a database, an API, files in S3, a flat file from a vendorRead yesterday's transaction CSV file from S3
TransformClean, reshape, enrich, and validate the data — fix data types, filter nulls, join to reference data, apply business rulesConvert date format, map region codes to region names, remove test accounts
LoadWrite the processed data to the target — a data warehouse table, a processed S3 prefix, a reporting databaseLoad cleaned records into Redshift's sales_fact table
Note: You may also hear ELT (Extract, Load, Transform) — this is common with cloud warehouses like Snowflake and Redshift where raw data is loaded first and transformation happens inside the warehouse using SQL. The distinction may come up in interviews.

What AWS Glue Does

AWS Glue is a fully managed ETL service. You don't need to set up servers — AWS handles the infrastructure. Glue has two main components a BA should understand:

Crawlers — Automated scanners that read data sources (S3 folders, databases, etc.) and automatically figure out the schema (column names, data types, partition structure). The crawler writes this metadata to the Data Catalog.

ETL Jobs — Scripts (written in Python or Spark) that define the actual transformation logic — reading from a source, applying transformations, and writing to a target. A data engineer writes these; as a BA you define the business rules they implement.

The Data Catalog

The Glue Data Catalog is a centralized metadata repository — think of it as a directory of all your data assets. It stores table definitions: what columns exist, what data types they are, where the data physically lives (S3 path or database connection), and how it's partitioned.

Once data is cataloged, it can be queried by Athena (without loading it anywhere) or referenced by Glue jobs. The Data Catalog makes data discoverable — you can search it to find out what data exists and where it is, which is directly relevant to BA work when scoping data requirements.

How a BA Interacts with Glue

ActivityWhat you do as a BA
RequirementsDefine the transformation rules in clear business language: "Null values in customer_id should be excluded," "Region code 'NA' maps to 'North America'," "Load date must be populated with the processing run date."
ValidationAfter a Glue job runs, check that the output in S3 or Redshift matches expected row counts, has no unexpected nulls, and reflects the business rules you defined.
UATRun test scenarios — what happens when a source file has a duplicate key? What happens with an empty file? Verify the pipeline handles edge cases correctly.
Data CatalogUse the catalog to understand what tables/data are already available before requesting new pipelines. Avoid duplicating work that already exists.

Common Interview Questions

Q: Can you explain what ETL means?
Sample answer: "ETL is Extract, Transform, Load — the process of pulling data from source systems, applying business rules and cleaning to transform it, and loading the result into a target like a data warehouse. For example, we extracted daily transaction files from an S3 landing zone, used a Glue job to validate and enrich the data, then loaded it into Redshift for reporting."

Q: Have you worked on data pipelines?
Sample answer: "Yes. I've been involved in defining requirements for Glue-based ETL pipelines — specifying source-to-target mappings, transformation rules, and data quality checks. I've also done UAT on pipeline output, validating that records counts and business rules were correctly implemented."

Q: What is a Data Catalog?

AWS Glue's Data Catalog is populated by crawlers. It makes data discoverable and is the first place to check when assessing what data is available for a new requirement."

Amazon Redshift

Cloud data warehouse · OLTP vs OLAP · How BAs use it · Interview Q&A

What is Redshift?

Amazon Redshift is a fully managed cloud data warehouse. It's designed to store large volumes of structured data and handle complex analytical queries fast — aggregating millions of rows, joining multiple large tables, running reports across years of data. You use standard SQL to query it, so if you know SQL, you can query Redshift.

Redshift is not designed for transactional use (inserting or updating individual rows frequently). It's built for analytics — read-heavy workloads where you're asking big questions across large datasets.

OLTP vs OLAP

This distinction comes up constantly in data warehouse interviews.

OLTP (Transactional)OLAP (Analytical)
Stands forOnline Transaction ProcessingOnline Analytical Processing
PurposeRecord day-to-day operations — orders, payments, bookingsAnalyze historical data — trends, aggregations, KPIs
OperationsFrequent INSERT, UPDATE, DELETE on individual rowsLarge SELECT queries across millions of rows

Example queryLook up one customer's current order statusTotal revenue by region for the last 3 years
Interview tip — "What's the difference between a database and a data warehouse?"
A database (OLTP) is optimized for recording transactions — it handles many small, fast reads and writes. A data warehouse (OLAP) is optimized for analytics — it handles fewer but much larger queries across historical data. You wouldn't run your production order system on Redshift, and you wouldn't run historical trend analysis on an operational SQL Server.

Distribution Keys and Sort Keys — High Level

These are Redshift performance concepts. You don't need to configure them, but you should know they exist and why they matter:

A distribution key determines how data is physically distributed across Redshift's nodes. If two large tables are joined on the same column and that column is the distribution key for both, the JOIN is much faster because the matching rows sit on the same node. Poor distribution key choices lead to slow queries because data has to be shuffled across nodes.

A sort key determines the physical order rows are stored on disk. If you frequently filter or sort by a date column, making it a sort key means Redshift can skip large chunks of irrelevant data. This is called zone map filtering.

As a BA, if queries are running slowly, these are things to ask the data engineering team about.

How a BA Queries Redshift

Redshift uses standard SQL with minor variations. You connect via a SQL client (like DBeaver, SQL Workbench, or the Redshift Query Editor in the AWS console) and write queries just as you would in SQL Server or Snowflake.

SQL 
-- Query a Redshift table — standard SQL syntax
SELECT region, SUM(revenue) AS total_revenue
FROM sales_fact
WHERE order_date >= '2024-01-01'
GROUP BY region
ORDER BY total_revenue DESC;

The COPY Command — Concept Only

COPY is the Redshift command used to bulk-load data from S3 into a Redshift table. It's far more efficient than INSERT for loading large volumes. You won't write COPY commands as a BA, but you'll hear about it when data engineers describe how data gets into Redshift from the S3 processed zone.

SQL 
-- Concept only — data engineers run this, not BAs
COPY sales_fact
FROM 's3://my-bucket/processed/sales/2024/'
IAM_ROLE 'arn:aws:iam::123456789:role/RedshiftS3Role'
FORMAT AS PARQUET;

Common Interview Questions

Q: What is Redshift and how is it different from a regular database?
Redshift is a cloud data warehouse optimized for analytical queries — large aggregations across historical data. A regular relational database like SQL Server is designed for transactional workloads — frequent small reads and writes on current data.

Q: What is the difference between OLTP and OLAP?
OLTP systems record transactions (your order system, your HR system). OLAP systems analyze that data at scale — think reporting, dashboards, trend analysis. Data typically flows from OLTP systems into OLAP systems via ETL pipelines.

Q: Have you worked with Snowflake? How is it similar to Redshift?
Both are cloud data warehouses designed for analytical workloads, support standard SQL, and scale independently. Snowflake separates storage from compute (making it easier to scale each independently). Both are commonly used as the query layer for BI tools like Power BI.

Practical tip: When asked about Redshift in an interview, connect it to the full data flow: "Data was extracted from our source systems, landed in S3, transformed by Glue, loaded into Redshift, and then Power BI connected to Redshift for reporting." That end-to-end story shows architectural awareness, which is exactly what they're testing for.
